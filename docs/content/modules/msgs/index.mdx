# Message Broker (`msgs`)

The `msgs` module is Coyote's message broker module. It supports three delivery modes: `stream`, `queue`, and `FIFO`.

Publishing is done to `topic`s that are shared across all delivery methods, with the different delivery methods only applying at consumption time. This means that you can consume the same messages in multiple different modes in parallel.

For example, let's say you built a hotel booking product and it has a stream of booking events for every time someone books. You may want to have a log of these events persisted to S3, but also have two different systems that need to act on each message: the email notification system, and the inventory processing system.

With Coyote, you can publish all of these to the `bookings` topic, and consume *the same messages* in parallel like this: (1) `stream` for S3 for a strictly ordered log, (2) `queue` consumer for the email notification system, and (3) `queue` consumer for the inventory processing system. Each consuming the same messages at their own pace, and every message consumed once by each different system.

Additionally, because Coyote is stream-backed in the background, you get all the benefits of a stream-backed system including: efficient writing, ordering, parallel-processing, and time-travel (acked `queue` messages aren't immediately deleted).

## Core concepts

### Messages

Message is the name of the basic unit that's sent over a topic. Each message has an associated payload (called `value`), a creation timestamp created by Coyote (`timestamp`), optional key-value headers (`headers`), and an optional `group_key` which is explained below.
Messages have an associated `timestamp` and.

### Topics

A `topic` is an ordered set of messages. This is where messages are published to.
Topics are created implicitly on first use, and you can have infinitely many.

### Consumer groups

A consumer group tracks how a set of consumers has progressed through a topic. When multiple consumers share a group, Coyote coordinates between them so each message is processed by exactly one member of the group; letting you scale out processing by adding more consumers without duplicating work.

Consumer groups are created implicitly on first use and are scoped to a topic + delivery mode combination.
Each delivery mode uses consumer groups slightly differently, so please refer to each delivery mode for how information on how to use them.

### Message group keys

Messages with the same `group_key` are processed in order for delivery modes that support it. So always set a key if ordering matters.

### Namespaces

A namespace groups topics together with shared retention and persistence settings.

## Delivery modes

As mentioned above, publishing is shared across all delivery methods, and the delivery methods only apply at consumption time.

### Queue

Queue mode is at-least-once delivery with configurable retry schedules and dead letter queues (DLQs).

Use a queue when you need a simple unordered task queue.

For more information, read [the queue docs](./queue.mdx).

### FIFO

FIFO is queue mode with per-key ordering: only one consumer can process each key at a time.

Use FIFO when you need ordered delivery that can scale to an arbitrary number of consumers.

For more information, read [the FIFO docs](./fifo.mdx).

### Stream

Stream mode is designed for globally ordered consumption with head-of-line blocking.

Use Stream when you need Kafka-like stream processing behavior which can't be achieved using FIFO. Otherwise, FIFO is probably preferred.

For more information, read [the stream docs](./stream.mdx).

The `group_key` behaves differently depending on the module:
- `stream`: messages with the same `group_key` are assigned to the same partition, which are in-turn always processed in order.
- `fifo`: messages with the same `group_key` are processed in order within the same consumer group, which means that if one consumer is currently processing a message with a specific key, no order consumers are going to get a message with the same key.
- `queue`: ignored, as queue has no notion of ordering.

## API reference

### Publishing

Publishing is the same across all delivery modes.

```json
// coyote.msgs.publish
{
  "topic": "orders",
  "msgs": [
    {
      "value": "the payload",
      "key": "user-123",
      "headers": { "content-type": "application/json" }
    },
    "a headerless message"
  ],
  "idempotency_key": "req-abc-123"
}
```

### Stream

#### coyote.msgs.stream.receive

```json
{
  "topic": "orders",
  "consumer_group": "G1",
  "batch_size": 100,
  "batch_wait_millis": 10000,
  "lease_duration_millis": 300000
}
```

Returns messages with their offsets and partition. The `topic` field in each message reflects the specific partition (e.g. `orders~0`), which you'll use when committing.

```json
{
  "msgs": [
    {
      "offset": 1,
      "topic": "orders~0",
      "value": "the payload",
      "headers": { "content-type": "application/json" },
      "timestamp": "..."
    }
  ]
}
```

#### coyote.msgs.stream.commit

Commits all messages up to and including the given offset for a partition.

```json
{
  "topic": "orders~0",
  "consumer_group": "G1",
  "offset": 42
}
```

#### coyote.msgs.stream.seek

Repositions a consumer group by timestamp or named position.

```json
{
  "topic": "orders",
  "consumer_group": "G1",
  "position": "earliest"
}
```

Accepts `timestamp` XOR `position` (`"earliest"` or `"latest"`). Seeking by offset requires targeting a specific partition (e.g. `orders~0`).

### Queue and FIFO

Queue and FIFO share the same API shape — substitute `fifo` for `queue` to use FIFO mode.

#### coyote.msgs.queue.receive

```json
{
  "topic": "orders",
  "consumer_group": "G1",
  "batch_size": 10,
  "lease_duration_millis": 30000
}
```

#### coyote.msgs.queue.ack

```json
{
  "topic": "orders",
  "consumer_group": "G1",
  "msg_id": "msg-abc"
}
```

#### coyote.msgs.queue.nack

```json
{
  "topic": "orders",
  "consumer_group": "G1",
  "msg_id": "msg-abc"
}
```

#### coyote.msgs.queue.extend-lease

Extends the processing lease on a message you haven't finished yet. Use this when you know you'll exceed the original lease duration.

```json
{
  "topic": "orders",
  "consumer_group": "G1",
  "msg_id": "msg-abc",
  "lease_duration_millis": 10000
}
```

#### coyote.msgs.queue.reset

Repositions a consumer group by timestamp or named position.

```json
{
  "topic": "orders",
  "consumer_group": "G1",
  "position": "earliest"
}
```

#### coyote.msgs.queue.redrive-dlq

Moves all messages from the DLQ back to the main topic for reprocessing.

```json
{
  "topic": "orders",
  "consumer_group": "G1"
}
```

### Configuration

#### coyote.msgs.topic.configure

```json
{
  "topic": "orders",
  "partitions": 8
}
```

#### coyote.msgs.queue.configure

Configure a retry schedule and DLQ for a specific topic and consumer group.

```json
{
  "topic": "orders",
  "consumer_group": "G1",
  "retry_schedule": [1000, 5000, 30000],
  "dlq": "orders-dlq"
}
```

If no consumer group configuration is set, a `nack` after all retries behaves like an `ack` (message is dropped).
