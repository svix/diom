# Message Broker (`msgs`)

The `msgs` module is Diom's message broker module. It supports three delivery modes: `stream`, `queue`, and `FIFO`.

Publishing is done to `topic`s that are shared across all delivery methods, with the different delivery methods only applying at consumption time. This means that you can consume the same messages in multiple different modes in parallel.

For example, let's say you built a hotel booking product and it has a stream of booking events for every time someone books. You may want to have a log of these events persisted to S3, but also have two different systems that need to act on each message: the email notification system, and the inventory processing system.

With Diom, you can publish all of these to the `bookings` topic, and consume *the same messages* in parallel like this: (1) `stream` for S3 for a strictly ordered log, (2) `queue` consumer for the email notification system, and (3) `queue` consumer for the inventory processing system. Each consuming the same messages at their own pace, and every message consumed once by each different system.

Additionally, because Diom is stream-backed in the background, you get all the benefits of a stream-backed system including: efficient writing, ordering, parallel-processing, and time-travel (acked `queue` messages aren't immediately deleted).

## Core concepts

### Namespaces

A namespace groups topics together with shared retention and persistence settings. Topics are created implicitly when you first publish or consume — no explicit topic setup required.

```json
// diom.msgs.namespace.create
{
  "persistence": "ephemeral",
  "retention": {
    "bytes": 4194304,
    "ms": 604800
  }
}
```

Topics can use any path structure: `events`, `orders/created`, `users/payments/refunds`, etc.

### Consumer groups

Consumer groups track where a set of consumers is in a topic. Like topics, they are created implicitly on first use. You can optionally configure a consumer group to set up retry schedules or a dead letter queue.

The same group name (e.g. `G1`) is independent across different topics.

### Partitions

Partitions are the unit of parallelism. A topic starts with one partition and can be scaled up with `diom.msgs.topic.configure`. Diom automatically assigns partitions to consumers and handles rebalancing transparently — you don't need to manage which consumer reads from which partition.

**Partitions are cheap.** If you expect to need parallel consumers, configure the partition count upfront. Increasing partition count later will break ordering for in-flight keys.

### Message keys

Setting a `group_key` on a message routes all messages with that key to the same partition, guaranteeing ordering within that key. Messages without a key are distributed round-robin.

**If ordering matters, always set a key.**

## Delivery modes

### Stream

Stream mode is designed for ordered, exactly-once consumption with support for parallel consumers. Each consumer group tracks its position via offsets per partition.

- **Ordered within a key** — messages with the same key always arrive in order.
- **Exactly-once** — commit offsets server-side after processing.
- **Parallel consumers** — Diom assigns partitions to consumers automatically; scale out by adding consumers.

Workflow: call `stream.receive` → process the batch → call `stream.commit` to advance the offset.

### Queue

Queue mode is at-least-once delivery with configurable retry schedules and dead letter queues (DLQs). One consumer holds a lease on each message at a time.

- **Retries on failure** — `nack` a message to retry it on the configured schedule.
- **DLQ support** — after retries are exhausted, messages are routed to a DLQ.
- **Lease-based** — if a consumer doesn't ack before the lease expires, the message is re-queued.

### FIFO queue

FIFO is queue mode with per-key ordering: only one message per key is in-flight at a time. If a message with key `user-123` is leased, no consumer will receive another `user-123` message until it's acked or the lease expires.

Use FIFO when you need ordered, at-least-once delivery without the full overhead of stream processing.

## API reference

### Publishing

Publishing is the same across all delivery modes.

```json
// diom.msgs.publish
{
  "topic": "orders",
  "msgs": [
    {
      "value": "the payload",
      "key": "user-123",
      "headers": { "content-type": "application/json" }
    },
    "a headerless message"
  ],
  "idempotency_key": "req-abc-123"
}
```

### Stream

#### diom.msgs.stream.receive

```json
{
  "topic": "orders",
  "consumer_group": "G1",
  "batch_size": 100,
  "batch_wait_ms": 10000,
  "lease_duration_ms": 300000
}
```

Returns messages with their offsets and partition. The `topic` field in each message reflects the specific partition (e.g. `orders~0`), which you'll use when committing.

```json
{
  "msgs": [
    {
      "offset": 1,
      "topic": "orders~0",
      "value": "the payload",
      "headers": { "content-type": "application/json" },
      "timestamp": "..."
    }
  ]
}
```

#### diom.msgs.stream.commit

Commits all messages up to and including the given offset for a partition.

```json
{
  "topic": "orders~0",
  "consumer_group": "G1",
  "offset": 42
}
```

#### diom.msgs.stream.seek

Repositions a consumer group by timestamp or named position.

```json
{
  "topic": "orders",
  "consumer_group": "G1",
  "position": "earliest"
}
```

Accepts `timestamp` XOR `position` (`"earliest"` or `"latest"`). Seeking by offset requires targeting a specific partition (e.g. `orders~0`).

### Queue and FIFO

Queue and FIFO share the same API shape — substitute `fifo` for `queue` to use FIFO mode.

#### diom.msgs.queue.receive

```json
{
  "topic": "orders",
  "consumer_group": "G1",
  "batch_size": 10,
  "lease_duration_ms": 30000
}
```

#### diom.msgs.queue.ack

```json
{
  "topic": "orders",
  "consumer_group": "G1",
  "msg_id": "msg-abc"
}
```

#### diom.msgs.queue.nack

```json
{
  "topic": "orders",
  "consumer_group": "G1",
  "msg_id": "msg-abc"
}
```

#### diom.msgs.queue.extend-lease

Extends the processing lease on a message you haven't finished yet. Use this when you know you'll exceed the original lease duration.

```json
{
  "topic": "orders",
  "consumer_group": "G1",
  "msg_id": "msg-abc",
  "lease_duration_ms": 10000
}
```

#### diom.msgs.queue.reset

Repositions a consumer group by timestamp or named position.

```json
{
  "topic": "orders",
  "consumer_group": "G1",
  "position": "earliest"
}
```

#### diom.msgs.queue.redrive-dlq

Moves all messages from the DLQ back to the main topic for reprocessing.

```json
{
  "topic": "orders",
  "consumer_group": "G1"
}
```

### Configuration

#### diom.msgs.topic.configure

```json
{
  "topic": "orders",
  "partitions": 8
}
```

#### diom.msgs.queue.configure

Configure a retry schedule and DLQ for a specific topic and consumer group.

```json
{
  "topic": "orders",
  "consumer_group": "G1",
  "retry_schedule": [1000, 5000, 30000],
  "dlq": "orders-dlq"
}
```

If no consumer group configuration is set, a `nack` after all retries behaves like an `ack` (message is dropped).
